# 콘텐츠 운영 가이드

## 콘텐츠 타입별 역할

| 타입 | 성격 | 비유 | 시간성 |
|------|------|------|--------|
| **Blog** | 의견·분석·논평 | 칼럼, 에세이 | 시점이 중요 (그때 그 생각) |
| **Guide** | 종합 레퍼런스 | 교과서, 핸드북 | 에버그린 (계속 업데이트) |
| **Study** | 학습 기록 | 수강 노트, 독서 노트 | 학습 과정 기록 |
| **Project** | 만든 것 | 포트폴리오 | 완성 시점 |

### 판단 기준

```
"X에 대한 내 생각/분석이 있다" → Blog
"X를 처음부터 끝까지 정리한 레퍼런스다" → Guide
"X를 공부하고 있다" → Study
"X를 만들었다" → Project
```

---

## Blog ↔ Guide 관계 원칙

**Blog = Why & What** (왜 중요한가, 무슨 일이 벌어지고 있는가)
**Guide = How** (어떻게 하는가, 상세 레퍼런스)

- 블로그에서 가이드 내용을 반복하지 않는다
- 블로그는 "읽을 이유"를 주고, 가이드로 보낸다
- 가이드는 블로그 없이도 독립적으로 읽을 수 있어야 한다

### 좋은 예

```
[Blog] "AI Native 팀이 왜 필요한가" — 현상 분석, 인사이트 2~3개, 가이드 링크
[Guide] "AI Native 팀 운영 가이드" — 6원칙, 4역할, 5안티패턴, 5단계 로드맵 (본체)
```

### 나쁜 예

```
[Blog] 6원칙, 4역할, 5안티패턴, 5단계 로드맵 전부 포함 (3000자+)
[Guide] 블로그와 동일 내용 반복
```

## Blog ↔ Blog 중복 방지

### 각 포스트는 하나의 고유한 테제(주장)를 가져야 한다

글 쓰기 전에 한 문장으로 테제를 정리한다:
- "AI가 콘텐츠 생태계 3축을 동시에 바꾸고 있다" (현상 분석)
- "AI에게 인용되려면 GEO 전략이 필요하다" (대응 전략)

테제가 한 문장으로 안 되면 범위를 좁힌다.

### 겹치는 데이터 처리

- 한쪽에서 상세히 다루고, 다른 쪽에서는 **인용 + 링크**
- 같은 현상의 다른 면을 다루는 경우, 서두에 관계를 명시한다:
  > "이전 글에서 X를 다뤘고, 이번엔 Y에 집중한다"

---

## Guide ↔ Guide 관계

가이드 간 관계는 명시적으로 연결한다:

```
agent (개론) → "패턴 심화는 agentic-patterns 가이드 참고"
agentic-patterns (심화) → "기초 개념은 agent 가이드 참고"
```

관련 가이드끼리 서로 링크를 걸어 독자가 자연스럽게 이동할 수 있게 한다.

---

## 새 글 쓸 때 체크리스트

1. **이미 비슷한 내용이 있는가?** — 기존 블로그/가이드 목록 확인
2. **이건 Blog인가 Guide인가?** — 의견/분석이면 Blog, 레퍼런스면 Guide
3. **기존 글과 겹치는 데이터가 있는가?** — 있으면 링크로 연결, 반복하지 않기
4. **이 글의 고유한 테제는?** — 한 문장으로 정리
