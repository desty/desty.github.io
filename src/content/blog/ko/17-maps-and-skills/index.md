이번 달 GitHub 트렌딩을 보다가 패턴이 하나 보였다. 비슷한 게 한꺼번에 올라왔다.

**codegraph**(5월 23일 트렌딩 2위, 24시간에 +2,400 스타), **Understand-Anything**, **ECC**(16만 스타), **pi**. 겉으로는 제각각이지만, 두 종류로 깔끔하게 묶인다. 하나는 **"AI에게 코드베이스 지도를 쥐여주는 도구"**, 다른 하나는 **"AI에게 스킬·도구를 갈아 끼우는 툴킷"**이다.

[지난 글](/blog/15-hermes-vs-openclaw/)에서 했던 것처럼, README 비교 대신 **네 개를 전부 클론해서 소스를 읽었다.** 이번엔 거기서 한 발 더 나가본다. 단순히 "뭐가 다른가"가 아니라 **"왜 지금 이게 떴는가"**, 그리고 클로드 코드·Codex·Gemini CLI·MCP가 움직이는 방향을 보면 **"다음은 어디로 가는가"**를 예측해본다.

결론부터: 코딩 에이전트 경쟁의 무게중심이 **모델·CLI에서 컨텍스트와 스킬로 넘어왔다.** 이 네 개가 그 신호다. 그리고 다음 전장은 — 글 끝에서 이야기하겠지만 — **신뢰**다.

---

## 왜 지금 ① : 에이전트는 코드를 "찾느라" 토큰을 태운다

코딩 에이전트를 써본 사람은 안다. 질문 하나 던지면 에이전트가 한참을 `grep`하고, 파일을 읽고, import를 따라가고, 호출 관계를 추적한다. **실제 답을 만들기 전에 "이 코드베이스가 뭐 하는 곳인지 파악하는" 단계** — 이걸 discovery phase라고 부른다. 토큰 예산의 상당 부분이 여기서 증발한다.

2026년 상반기 들어 이게 더 아픈 문제가 됐다. 세 가지가 겹쳤다.

- **토큰 효율이 경쟁 축이 됐다.** OpenAI는 Codex CLI가 클로드 코드보다 토큰 효율이 약 4배라고 주장한다. 같은 작업을 누가 더 싸게 하느냐가 마케팅 포인트가 됐다는 뜻이다.
- **컨텍스트 창은 1M으로 커졌지만, 그게 공짜는 아니다.** Gemini도 Opus도 100만 토큰을 넣을 수 있다. 하지만 매번 코드베이스를 통째로 욱여넣는 건 느리고 비싸다.
- **MCP가 표준으로 수렴했다.** Anthropic·Microsoft·OpenAI가 Model Context Protocol을 공통 인터페이스로 받아들이면서, "한 번 만든 도구를 모든 에이전트에 꽂는" 게 가능해졌다.

그래서 나온 답이 **"코드를 미리 지도로 만들어 두자"**다. 매번 더듬지 말고, 인덱싱해 둔 구조를 질의하자. codegraph와 Understand-Anything이 정확히 이 지점을 노린다. 그런데 **'지도'를 만드는 방식이 정반대다.**

## 코드 지도 2종 : codegraph vs Understand-Anything

### codegraph — 기계가 검증한 지도

**정체.** tree-sitter로 코드를 파싱해 함수·클래스·import·호출 관계를 추출하고, 이걸 로컬 **SQLite 그래프**에 넣는 도구다. AI 에이전트는 이 그래프를 **MCP 서버**를 통해 질의한다. `codegraph_search` / `callers` / `callees` / `trace` / `impact` 같은 10개 도구가 노출된다.

**핵심 기술.**
- 파싱은 tree-sitter WASM 문법(레포의 `.wasm` 파일들이 그것이다). 저장은 Node 내장 `node:sqlite` + FTS5 풀텍스트 인덱스.
- **LLM이 전혀 들어가지 않는다.** 임베딩도, 모델 호출도 없다. 순수 정적 분석이다. 그래서 그래프의 내용은 코드에서 기계적으로 도출된 사실이다 — 환각이 끼어들 여지가 없다.
- 코드를 직접 뜯어 확인한 결과 **"100% 로컬, 외부 전송 없음"은 사실이었다.** 외부 HTTP·텔레메트리가 전무하고, 유일한 네트워크 사용은 로컬 데몬과 통신하는 유닉스 소켓 IPC뿐이다.

**숫자는 가려 들어라.** 레포는 "토큰 57%, 도구 호출 62% 절감"을 내세운다(제3자 소개글에선 토큰 59%·도구 70%, 심지어 package.json엔 94%라는 숫자도 있다). 벤치마크 하니스가 실제로 들어 있긴 한데 **저자 자체 측정에 표본이 작다(중앙값 n=4, 일부는 n=1).** 문서마다 숫자가 30~94%로 들쭉날쭉하다. **방향(줄어든다)은 믿되, 특정 퍼센트는 마케팅으로 취급하는 게 안전하다.**

> **언제 쓰면 좋은가**
> 큰 코드베이스에서 에이전트가 헤매며 토큰을 태우는 게 보일 때. 클로드 코드·Codex·Cursor 어디든 MCP로 꽂아 쓰고 싶을 때. 외부로 코드가 한 줄도 나가면 안 되는 환경(사내·보안)일 때. **에이전트의 연료**로 쓰는 지도다.

### Understand-Anything — 사람이 읽는 지도

**정체.** 같은 "코드 → 지식 그래프"지만 결과물도 용도도 다르다. **LLM 멀티에이전트 파이프라인**이 프로젝트를 훑고, 요약하고, 아키텍처 레이어로 묶어서 `.understand-anything/knowledge-graph.json` 한 파일로 만든다. 그리고 **React Flow 기반 인터랙티브 대시보드**로 시각화한다 — 레이어별 색상, 클릭하면 코드·관계·평문 설명, 심지어 보는 사람(주니어/PM/파워유저)에 따라 상세도가 달라진다.

**핵심 기술.**
- tree-sitter로 파일·import 같은 뼈대는 기계적으로 긁지만, **요약·레이어·관계·투어는 LLM이 작성한다.** 그래프의 7단계 빌드 과정이 `SKILL.md`에 프롬프트로 정의돼 있고, 호스트 에이전트가 서브에이전트를 돌려 채운다.
- 즉 이 그래프는 **"LLM이 써준 JSON"**이다. 코드가 검증한 사실이 아니라, 모델이 코드를 읽고 해석한 결과다. 품질은 구동 모델에 달려 있다.
- 솔직히 짚자면, 소스 안에 시맨틱 검색 엔진(`embedding-search.ts`)이 있지만 **임베딩을 생성하는 곳이 파이프라인 어디에도 없다.** 실제 "질문하기"는 JSON을 grep하는 수준이다. "시맨틱 검색"은 다소 과장이다.
- Claude Code·Codex·Cursor·Copilot·Gemini CLI 대응은 진짜지만, **MCP가 아니라 설정 파일 + 마크다운 스킬 심볼릭 링크** 방식이다. 텔레메트리는 없다.

> **언제 쓰면 좋은가**
> 낯선 코드베이스에 새로 합류해서 **사람이 구조를 빠르게 이해해야** 할 때. 팀 온보딩, 레거시 파악, "이 도메인 로직이 어떻게 흐르지?"를 그림으로 보고 싶을 때. 코드뿐 아니라 문서·위키(예: Karpathy LLM 위키)도 그래프로 만든다. **사람을 위한 지도**다.

### 한 줄로 갈리는 지점

| | codegraph | Understand-Anything |
|---|---|---|
| 그래프 생성 | tree-sitter 정적 분석 (LLM 0%) | LLM 멀티에이전트 파이프라인 |
| 결과물 | SQLite + FTS5 (질의용) | JSON + 시각화 대시보드 (열람용) |
| 신뢰 모델 | 기계가 검증한 사실 | 모델이 해석한 결과 |
| 소비 주체 | **에이전트** (MCP 10개 도구) | **사람** (브라우저 대시보드) |
| 통합 방식 | MCP 네이티브 | 설정파일 + 스킬 심볼릭 링크 |
| 핵심 가치 | 토큰·도구호출 절감 | 구조 이해·온보딩 |

둘은 경쟁자가 아니다. **codegraph는 에이전트가 먹는 연료고, Understand-Anything은 사람이 보는 지도다.** 이름이 비슷해서 같은 카테고리로 묶이지만, "지도를 누가 읽느냐"가 다르면 설계가 전부 갈린다. 코드를 열어보기 전엔 안 보이는 차이다.

## 왜 지금 ② : 스킬이 크로스 에이전트 표준이 됐다

두 번째 묶음으로 넘어가자. 이쪽 배경은 더 단순하다.

`SKILL.md` — YAML 프론트매터에 "이름 + 언제 쓰는지"를 적고 본문에 절차를 쓰는 단순한 포맷 — 이 **사실상 표준이 됐다.** 클로드 코드에서 만든 스킬이 Codex CLI, Cursor, OpenCode 등 SKILL.md 호환 에이전트에서 그대로 돈다. Anthropic이 공식 `skills` 레포와 직종별 `knowledge-work-plugins`로 모범을 깔았고, 마켓플레이스가 폭발했다. 어떤 디렉토리는 스킬 6,700개, 마켓플레이스 2,500개를 자랑한다.

표준이 생기고 진입장벽이 낮아지면 **골드러시**가 온다. "스킬을 최대한 많이 모아둔 번들"과 "스킬·에이전트를 돌리는 인프라" 양쪽으로 사람이 몰린다. ECC와 pi가 정확히 그 양극단이다.

## 스킬·툴킷 2종 : ECC vs pi

### ECC (Everything Claude Code) — 양으로 덮는 설정 레이어

**정체.** 클로드 코드·Cursor·Codex·OpenCode·Gemini·Zed·Copilot까지 여러 하네스에 한꺼번에 꽂는 **거대한 설정·스킬 번들**이다. 2025년 9월 Anthropic 해커톤 우승에서 출발했고, 제3자 집계 기준 **16만 스타**를 넘었다. 에이전트 약 60개, 스킬 약 250개, 명령어 76개, 훅·보안 스캐너(AgentShield)를 묶어 버전 관리한다.

**핵심 기술 — 그리고 솔직한 코멘트.**
- 멀티 하네스 통합은 진짜다. `.claude/` `.cursor/` `.codex/` `.gemini/` 등 하네스별 디렉토리와 어댑터, `.mcp.json`에 6개 MCP 서버, 동작하는 훅 디스패처가 들어 있다. 메모리도 파일 기반이지만 실제 구현체(세션 훅, 1,800줄짜리 `instinct-cli.py`)가 있다.
- **다만 코드 밀도는 따져봐야 한다.** 직접 세어보니 **스킬 약 250개 중 실제 코드가 든 건 12개뿐, 나머지는 마크다운 프롬프트다.** 관세·비자·헬스케어 같은 코딩과 무관한 버티컬도 잔뜩 섞여 있다. "보안"도 상당수는 75개 파일에 복붙된 프롬프트 방어 문구이고, 핵심 보안 모니터는 남의 pip 패키지를 감싼 래퍼다(기본 꺼짐).
- 스타 수는 압도적이지만, **개수 = 가치는 아니라는 점**을 코드가 보여준다. 이건 ECC를 깎으려는 게 아니라, 트렌딩 번들을 평가하는 일반 원칙이다.

> **언제 쓰면 좋은가**
> 여러 에이전트를 오가며 쓰는데 **설정을 한 벌로 통일**하고 싶을 때. 훅·MCP·하네스 어댑터를 처음부터 짜기 싫고, 잘 차려진 출발점이 필요할 때. 단, 250개 스킬을 다 쓴다는 환상은 버리고 **필요한 12개를 골라 쓰는** 마음가짐으로.

### pi — 깊이로 파는 인프라 레이어

**정체.** libGDX를 만든 **Mario Zechner**가 만들어 2026년 4월 Earendil에 합류하며 가져온 **TypeScript 모노레포 툴킷**이다. 스킬 더미가 아니라, 에이전트를 **돌리는 기반**을 짠다. 네 개 패키지로 구성된다.

| 패키지 | 역할 |
|---|---|
| `pi-coding-agent` | 코딩 에이전트 CLI |
| `pi-ai` | 여러 LLM을 묶는 통합 API |
| `pi-agent-core` | 에이전트 실행 런타임 (도구 호출·상태 관리) |
| `pi-tui` | 터미널 UI 라이브러리 |

**핵심 기술.**
- **통합 LLM API.** Anthropic·OpenAI(+Codex/Azure)·Google(Gemini/Vertex)·Mistral·AWS Bedrock까지 9개 제공자를 한 인터페이스로 추상화한다. 벤더 락인 방지가 설계 목표다.
- **공급망 하드닝이 진짜다.** 코드로 확인했다 — 외부 의존성은 전부 정확 버전 핀(`save-exact`), 배포 패키지엔 134개 무결성 해시를 박은 shrinkwrap, 설치 시 `--ignore-scripts`, 핀 검사 CI 스크립트까지. "공급망 안전 중시"가 슬로건이 아니라 코드에 박혀 있다.
- 엔지니어링 성숙도가 높다. Biome, 엄격한 TS, 테스트 252개. (참고: Slack 봇·vLLM 연동은 토킷 전체 범위엔 있지만, 메인 모노레포가 아닌 별도 레포(`pi-chat`)에 있다.)

> **언제 쓰면 좋은가**
> 에이전트를 **직접 만들 때.** 여러 LLM을 갈아 끼우는 앱을 짜거나, 자체 코딩 에이전트/TUI를 올릴 때. 특히 **공급망 보안이 중요한 조직**이라면 핀·shrinkwrap·ignore-scripts가 그대로 참고서가 된다. 끝단 사용자용 번들이 아니라 **빌더용 기반**이다.

### 한 줄로 갈리는 지점

| | ECC | pi |
|---|---|---|
| 정체 | 멀티 하네스 설정·스킬 번들 | 에이전트 개발 인프라 모노레포 |
| 무게중심 | **양** (에이전트·스킬 개수) | **깊이** (런타임·통합·공급망) |
| 대상 | 에이전트를 **쓰는** 사람 | 에이전트를 **만드는** 사람 |
| 신뢰 근거 | 스타 수·해커톤 출신 | 코드·공급망 하드닝·저자 이력 |
| 함정 | 개수 인플레 (코드 든 스킬 12/250) | 끝단 사용자에겐 과한 추상화 |

## 그래서, 다음은 어디로 가는가

네 개를 코드까지 읽고 최근 플랫폼 흐름과 겹쳐 보면, 세 가지 방향이 보인다.

**① 코드 인덱싱은 에이전트 안으로 흡수되거나 MCP 표준으로 수렴한다.**
지금은 codegraph 같은 별도 도구가 트렌딩을 타지만, "에이전트가 코드베이스를 더듬느라 토큰을 태운다"는 문제는 너무 보편적이라 결국 **에이전트 자체가 내장하거나, MCP 표준 도구로 빨려 들어갈** 가능성이 높다. 지금의 스탠드얼론 인덱서들은 과도기 베팅이다 — 표준이 자리 잡기 전 빈틈을 메우는.

**② 스킬은 커머디티화되고, 해자는 "개수"에서 "큐레이션·검증"으로 옮겨간다.**
스킬 6,700개 시대에 "우리는 250개 있다"는 더 이상 자랑이 아니다. ECC의 코드 밀도가 보여주듯, **양으로는 변별이 안 된다.** 다음 단계는 "이 스킬이 실제로 동작하는가, 검증됐는가"다. 통과 못 하는 무결성 게이트로 인용을 검증하고 테스트 1,500개를 돌리는 [academic-research-skills](https://github.com/Imbad0202/academic-research-skills) 같은 접근이 — 화려하진 않아도 — 더 미래에 가깝다.

**③ 다음 전장은 신뢰다.**
에이전트가 더 자율적으로, 더 많은 코드를 실행할수록 **비용보다 신뢰가 중요해진다.** 세 갈래로 온다.
- **공급망** — pi가 핀·shrinkwrap·ignore-scripts로 미리 답한 그 문제. 에이전트가 npm 패키지를 깔고 코드를 실행하는 세상에서 이건 선택이 아니다.
- **검증** — 그래프가, 스킬이, 답이 "진짜"인가. codegraph가 LLM을 빼고 정적 분석만 쓴 것도, 환각 없는 사실을 주려는 신뢰 설계다.
- **출처** — 트렌딩엔 이름만 그럴싸한 것도 많다(이번에 같이 본 것 중엔 "Anthropic"을 앞에 붙였지만 개인이 만든 비공식 레포도 있었다). 이름이 아니라 코드와 저자로 판단해야 한다.

## 마치며

이번에 본 네 개는 전부 **단일 squash 커밋**으로 클론됐다. 트렌딩 신상이라 개발 이력이 안 보인다는 뜻이다. 별 개수도, 스킬 개수도, 이름도 — 표면 신호는 다 흔들린다. 결국 [지난 글](/blog/15-hermes-vs-openclaw/)에서도, 이번에도 같은 결론에 닿는다. **코드를 열어봐야 진짜가 보인다.**

[블로그 #14](/blog/14-agent-engineering/)에서 에이전트 엔지니어가 진짜 설계하는 건 모델 바깥의 레이어들이라고 썼다. 이번 트렌딩이 그걸 그대로 보여준다. 모델은 이미 강력하다. 사람들이 지금 만드는 건 **그 모델에게 코드를 어떻게 먹이고(컨텍스트), 무슨 일을 시키고(스킬), 어떻게 믿을 것인가(신뢰)**다.

다음 주 트렌딩을 또 보게 될 것이다. 그때 던질 질문은 하나다.

**이건 새로운 능력인가, 아니면 같은 능력을 더 많이 쌓아 올린 더미인가?**
